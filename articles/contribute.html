In this article, complete instructions on how to practically go about making a contribution to the wiki will be documented. For the foreseeable future, it will be required by an author to get aquainted with the particular way that the website is designed and styled. Be aware that these particular solutions might not always be optimal, or even very good, from a conventional perspective. This does leave a great deal of room to make contributions that improve on these rag-tag, homemade design decitions. If you find something on this wiki that is crap and you have a good idea on how it might become less so, then keep reading this article and submit your suggestions.

EXTERNAL RESOURCES

The wiki is compeletely stored on a Git repository, meaning any change is done through pull requests. This offers an existing open source framework for managing, moderating and administrating the wiki without any need for user accounts. The repo is currently publicly hosted on Github and you'll find it right here.

For managing active maintenance and tasks pertaining to the wiki, there's a publicly readable Trello board which can be used by anyone to view what sort of improvements have already been identified and what is being worked on.

If you would like to grab an improvement task for yourself, or perhaps suggest one, an email is the way to go. Just shoot an email to hjalle@freshmanbasix.com. If interest ever grows large enough, there might appear a discord server as well. Time will tell.

DIRECTORY HIERARCHY

Under the root, the wiki is organized into a few simple folders. The complexity remains quite low which allows for the website directory to consist only a small number of files.

root/-- |-index.html |-style.css |-style/- | |-stylesheet_category.css |-CNAME |-README.md |-scripts/- | |-script.js | |-rsc/- | |-article- | |-section- | |-file.format | |-articles/- |-article.html
  • index.html is the core of the website, being loaded as the default on a web request. This file should ONLY contain the primary framework of the website, leaving the actual content distributed elsewhere.
  • style.css is the general stylesheet, linked in index.html. This file imports all members under the style/ directory, which individually contain the complete styling ruleset, categorized by function.
  • README.md is a github specific file that only corresponds to the github repo Readme section.
  • CNAME is for the CNAME record at the domain name registrar.
  • All js scripts are located under the scripts/ directory. Remember that each added scripts must be imported with a script element in index.html.
  • The articles/ folder contain the primary content of each article. The article IDs correspond to the name of the file exactly, subtracting the file ending.
  • Every file, image or other visual resource useed somewhere in the wiki should be stored under the rsc/ directory. Create subdirectories in the suggested manner to avoid long file names and be sure to name the file in a consise and clear way.

HTML MAP OVERVIEW

The core content of the wiki is contained within a CSS flexbox dividing itself into 3 columns where the central columns contains the main article and is 4 times as wide as the outer ones.

NAVIGATION BAR

The leftmost column offers an overview and navigation of articles. Under the nav element with id="nav-bar" is an unordered list, with each li formated as follows:

<li class="nav-item"> <a href="#article-route">ARTICLE-NAME</a> </li>

Each list-item must have the class attribute set to nav-item and contain an anchor element with the href attribute set to call the javascript function openArticle().

WIKI ARTICLE COLUMN

The article itself is contained within the main element with id wiki-column. The whole article itself is stored inside a article element which has an id value corresponding to the article file name, as found in the /articles directory. It also has the class value wiki-entry. In general, the article is divided into several section elements, the first of which also contains the table of contents. Below follows a diagram along with core example code to illustrate how the hierarchy of a article must be structured to render properly.

<article id="file-name" class="wiki-entry"> <h1 class="column-header">Title</h1> <section class="sub-section"> <div id="toc-container"> <h1 class="column-header toc-header">CONTENT</h1> <ul class="toc"> <li><a href="#sub-section-1">Sub Header</a></li> </ul> </div> <p>Introduction</p> </section> <h2 id="#sub-section-1">Sub Header</h2> <section class="sub-section"> <p> Sub section content </p> </section> </article>

Note that each h2 header preceeds the section element for each sub-section. This was chosen because it made it simpler to reliably make the margins larger for the text content than the header itself. It does feel slightly unintuitive though, so an alternate solution is being worked on.

FORMATING AND STYLE-RULES

The styling in the wiki is built on the assumption that all articles are written and formated in a consistent way. It does not make use of any kind of third party documentation framework or software. It is simply done completely by hand. There are a number of predefined blocks that can be used to present information in a effective way. Below follows the available elements that can be used, along with descriptions and examples.

All predefined components listed below make up ALL components with pre-existing styling that fit consistently. Of course, anything not mentioned should fit just fine and may be used freely. The components that are listed however MUST be written that way for consistency of presentation.

A few words should be said about display style and nesting before going further. Elements that are not explicitly intended by the default HTML style to be inline elements should not be used as such. It is the recomended practice to add content one component at a time. The exception being lists. Inside the <li> tag, any component can essentially be added in series to achieve an effective slight increase in margin size.

Note that each<li> elements do not have to contain a paragraph element. Only this list items containing multiple paragraphs will need the content wrapped further.

REGULAR BLOCKS

Two levels of Sub-sub-sections are supported by the css styling. h3 and h4 elements are all that are needed.

The plain <p> element is all you need to create a correctly styled Paragraph. Each paragraph of text need an element on its own. Manual linebreaks inside p elements are to be avoided

To mark out points where future TODOs should be added, there is a dedicated <p> element with the class attribute set to addme. It will render a very eyecathcing and pretty ugly object. Use them mainly to quickly mark out the general idea of what should be added.

<p class="addme"></p>

INLINE BLOCKS

Basic boldface and italic text can be used with the native <b> tag for plain boldface and <em> is for cursive text.

Math has dedicated styling objects, see the math styling section below

For specialized styling, there are earmarked styling rules that can be used with a span element to achieve particular inline blocks. For marking out programming snippets or general code-adjacent terms, use the example below.

<span class="inline code"> Place inline code here </span>

For general keywords or concepts that have special importance and need to stick out of the paragraph, use the following example.

<span class="keyword"> Place inline keyword here </span>

EMPASIZED BLOCKS

  • Blockquotes can be created in a styled fashion by using the <blockquote> element inside i <figure> element along with a added <figcaption>.

    <figure class="blockquote"> <blockquote cite="optional link to source"> <p>Actual quote</p> </blockquote> <figcaption>- source, <cite><a href="link to source">link text</a></cite></figcaption> </figure>

  • Figures & Images are added in a Latex-esque manner, using the <figure> element as an environment, the <img> as the image source, and lastly the <figcaption> element used to pop on a nice image caption.

    <figure> <img src="filepath" alt=""> <figcaption>Caption</figcaption> </figure>

    Take note that for any diagram the caption should include a link to a xml file that can be used to edit the image itself without startng from scratch. Source files are all stored in the /rsc directory.

  • Code snippets are simple boxes with example code. A <div> element with the class attribute set to "box snippet code" will render the box correctly. Be aware that the code cnippet box renders ALL text as written in the html file, which includes the indentation. So if you don't push it back to the beginning of the line manually, it's going to look bonkers.

    <div class="box snippet code">Put your example code here and it will be rendered like this box</div>

  • Attention blocks are a good way to make some very important information stick out like a smack in the face.

    <div class="box attention">Put something extra important here...</div>
    ... and the info will render like this

TABLES

Tables are currently only supported in two-column form, with each column eating up 50% of the width. The most appropriate use is to simply make visually appealing lists of resource and link pairs.

<table> <caption>top caption</caption> <colgroup span="2"></colgroup> <tr> <td>description</td> <td>link</td> </tr> </table>

ATTRIBUTION

Any and all resources fetched from somewhere else need to be clearly attributed, prefereably in direct proximity to the material itself. The most appropriate method is to a variant of the Harvard model. It is best to introduce the source and link it properly so that the reader understands clearly how the reference relates to the discussed material. For standard literature references, author and year as a hyperlink will do.
