DOCUMENT OBJECT MODEL

Being a markup language, HTML is very much oriented towards generating documents. That history means that the term document has come to essentially refer to the root of any webpage. From the perspective of the network, the actual html file generating a webpage is called the document. In terms of scope, the document is the absolute root. The Document Object Model, DOM, is the formalized data representation to abstract all objects within a document. Each document can be comprised of one or more actual HTML files, so it can be important to point out that the document in definite form actually mean some kind of rendered web resource.

The DOM abstracts a document into a hierarchical tree structure where each object is represented by a node. This does not only include the elements of the core file, but also multiple other attributes, properties and other stuff. The DOM can be discussed for hours, but some very useful reading can be done by beginners at the Document Object Model Wikipedia Page. Furthermore, one might want to take a look at the MDN Introduction to the DOM pages.

The most important factors to keep in mind is that the DOM is what specifies how to access and modify the content of a document. It's not extremely essential to understand in depth as an actual model apart from the fact that it is what formalizes the conceptual tree structure. It does establish a large set of rules though, so anyone interested in becoming a very skilled web developer will want to have a thorough theoretical and practical understanding of the DOM.

HYPERTEXT MARKUP LANGUAGE

God only knows what's so damn hyper about it, but the language is a syntax for marking up data to enable its presentation to be rendered by some other software. The complete language set is very small, consisting of a few central components making up the syntax glossary. There's just a few of them, illustrating how quick HTML truly is to review and start using

SYNTAX GLOSSARY

  1. First out is the HTML ELEMENT, exemplified below with a <img> element. This is the core building block if HTML, consisting of two TAGS, much like a set of parenthesees, who OPEN and CLOSE the element.
    <article class="wiki-article"> </article>
    Generally browsers can work around omitting the closing tag depending on the circumstances. It is usually best practice to thoroughly add both tags in order to avoid undefined behavior. Some elements are intended to hold no content, known as Empty Elements. These usually even require the closing tag to be left out.
  2. Secondly comes the ATTRIBUTE. These are descriptors defining the meta data and presentation of the Element. Attributes are always located inside the opening tag and separated by white spaces. Generally speaking, most attributes use in practice are specific for the element. Styling is possible through attributes, many CSS properties have attribute counter-parts, but it's a BAD IDEA! Two very important attributes are class and id which are both used to organize the HTML document and allow more intricate behaviour and style. Both CSS and JS make good use of classes and id's to effectively parse and access specific content for modification and interaction.
    class="wiki-article"
  3. Any element will have ONE PARENT and ANY NUMBER of CHILDREN, i.e one element of which said element is inside and zero or more elements in between the tags.
    <figure> <img src="/img/file.png" alt="text"></img> <figcaption>Here's a caption</figcaption> </figure>

CORE DOCUMENT ANATOMY

From the history that gave birth to it, HTML is naturally designed to generate a typical paper document conventionally structured like an academic essay or technical litterature. That means it has predefined pieces that can make up sections, headers, paragprahs and so on. This is because that was the intended use at first. With webpage design becoming a lot more colorful and ballsy, the old model doesn't quite fit anymore. Still, there have never been any realistic competitors and HTML was just so established by the time that internet speeds where high enough to support anything other than confetti colored walls of text that developers just made do. If anything, modern frameworks have been constructed on top of the HTML/CSS/JS combo to provide tools making current design a bit more streamlined. At the core though, they are all still just generating HTML.

Any HTML element will create a node in the DOM. Any natural hierarchical order of elements is only infered, with a fe exceptions. Syntactically, the browser isn't going to care that you place a <main> element inside a <h1> element. By default, it's going to look pretty weird, but as long as the diagram of nodes can be drawn, the page will work. The attributes are not the same for all elements of course. You can't make a <p> element behave like an <img> element. This is true for styling properties as well, but in most cases, you can make one element look and feel like another. The reason we have multiple types of elements are to allow us to:

  1. Embed external content, i.e images and the like
  2. Parse the document in a useful way
  3. Define consistent styling rules

For a HTML document to be legally recognized by a browser, it will need to contain a couple of core components. The DOCTYPE tag establishes what kind of document it is. The version of HTML can be defined in this element. The <html> element establishes the root node of the document. The root must have two children, the <head> and <body> elements.

The example below is a good standard template for creating a basic html which imports a stylesheet from a separate file, which is the only sensible way to do things.

<!DOCTYPE html> <html lang="en" dir="ltr"> <head> <meta charset="utf-8"> <title>Website Title</title> <link rel="stylesheet" type="text/css" href="style.css"> </head> <body> <!-- Content of the webpage --> </body> </html>

Element Blocks

The last important thing to cover is how html elements are placed in relation to each other. There are essentially two main important ways that a element can be displayed:

  1. As a series of blocks that are placed beneath each other.
  2. As inline objects, essentially putting the element on the same line as the previous one.

There are several other display modes for html, mainly flex and grid, which will be touched upon in the CSS section. But at it's core, html is all about placing blocks of content in the correct place. This whole subsection touches on style to a large degree, but is still a central concept of how to structure HTML. Most elements will be displayed as a block by default. Headers and paragraphs are examples of this. They will simply render one after the other on a new lines. Meanwhile, the elements marking up emphasis are all displayed on the same line as the context they are situated in.

This will be sufficient for any reader to move forwards with a sufficient understanding to start reading further on the topic.

CASCADE, SPECIFICITY AND INHERITANCE

There are three concept innate to CSS that are very important to grasp when working effectively with style sheets. These are Cascading, Specificity, and Inheritance. In essence, a CSS file is a long set of rules, defining how elements are to be rendered. These rules follow a general behaviour which will be discussed in this section, but can should be studied in much further detail by those who wish to be very skilled front end developers.

The cascading part of the name comes from the fact that the rules are applied on a sequential cascading order, where each styling rule is read at a time, from top to bottom, and applied linearly. In short, this means that rules appearing later in the file will potentially overwrite the ones appearing earlier.

Whether or not this conflict will occur is highly dependent on the specificity of the rule, i.e exactly which elements it will apply to. Depending on the complexity, rules might have to be very intricateky designed to apply as desired. This puts a responsibility on the designer to establish a very clear structure on the document as a whole so that rules can be applied in a systematic way.

The last major concept is that of Inheritance. All CSS rules work by defining properties. Some properties can be inherited from parent elements. Some apply only to the element itself and some will affect all its children. Knowing which is which is very important to achieve the intended outcome.

THE RULESET

While HTML was a very simple language, the syntax of CSS is even more simple, even by far. There are two ways to include CSS styling in a HTML document:

  1. Like a schmuck, as the content of a <style> element, which must be the direct descendant of the <head> element.
    <head> <style> #some rules </style> </head>
  2. Professionally, as the content of a separate file, linked into the html file with a <link> element. Multiple CSS files can be linked into the same HTML file but the cascading behaviour will apply.
    <link rel="stylesheet" type="text/css" href="style.css">

Below is an example of what the content of a CSS file might look like. The sample code below is an example of what is called a ruleset, or in short just a css rule.

body { width: 80vw; min-width: 800px; margin: 0 auto; margin-bottom: 10vh; }

In this case, what is shown are all the styling rules that will apply to any body elements. The constituent parts of a ruleset are broken down below and syntax.

Note that each singular rule separates property and value with a :, and ends each rule with the ; character.

Formally speaking, the syntax rules for rulesets are as follows. Note that any rule that is incorrectly formulated is simply ignored. One very important consequence of this is that a missed semicolon, ;, could possibly throw the syntax off enough to make the browser discard the entire ruleset.

ruleset ::= selectors-list { properties-list } selectors-list ::= selector[:pseudo-class] [::pseudo-element] [, selectors-list] properties-list ::= [property : value] [; properties-list]
  • The Selector is the part specifying which elements that the ruleset apply to. Details of the many kinds of selectors can be at the CSS Reference. It is possible to vary the level of specificity a great deal in terms of place of decendancy, siblings, class affiliation etc. Being precise with the selector becomes increasingly important as the complexity of style grows. Selectors are what allows the web developer to assign multiple layers of style and should be understood thoroughly.
  • Properties are in this case differing style properties that can be modified to change the presentation of the element. Some properties correlate directly to HTML attributes and current best practice is to only use CSS to modify them. Most properties are shared by all elements, with a few being specific for certain element types. A complete list of all properties can be found indexed here.
  • Depending on the property itself, the Property Value comes in a multitude of data types. To see exactly what values are accepted for a given property it is best to refer to the documentation. MDN has a good reference sheet for the CSS data types here.
  • Every rule is made up of a Declaration, defined above as properties-list. Any ruleset can essentially contain as many declarations as there are properties supported by the selector in question.

For readability it is always a good idea to put a single rule per line, but the syntax does not require a linebreak. Be sure to also keep in mind that the order of the rules make all the difference. All changes applied by some rule at the bottom will overwrite a rule at the top.

Another important concept relevant to properties are the Shorthands, which are composite properties that combine multiple properties in one and allow us to define multiple values in a sinlge line separated by a space character

margin: 1px 2px 1px 2px;

The example above shows how one might define the values of all the margin properties in one. Each value, starting from the left, represent a clockwise movement from top, right, bottom and lastly left. Some shorthand support several combinations, for instance margin can take two values, which then applies to top/bottom and left/reft respectively. Check out MDN for a reference and more information.

Variables and Environment

Variables can be very effective in structuring a large ruleset, especially when establishing a color palette. Variables can be declared and defined in a separate ruleset with the :root pseudo-class selector. This establishes the variable in the global scope and lets any ruleset access it.

:root { --palette-orange: #f2951d; }

All variables must begin with a double hyphen, --. Variables can be declared inside any ruleset, but will only exist in that scope.

In order to use the variable the var() function is used.

element { background-color: var(--palette-orange); }

On top of custom defined variables, there are also some very practical environment and language defined variables that can be practical to know.

  • The viewport is the window used to render the website. The value of this variable is dependent on the client side screen as well as the actual window rendering the content, but can be accessed directly in the CSS code in several ways. JAVASCRIPT

Javascript is an entire god damn language. It was designed to improve the functionality of web pages and has a ton of quirks, but its a full feature language nonetheless. Apart from a few extremely basic starter tips, this wiki will only refer to official documentation. Some excellent reading resources can be found at the MDN JavaScript pages. One very useful book that might be of interest is JavaScript: The Definitive Guide by David Flanagan (O'Reilly Media). It is utterly pointless trying to establish any foundational knowledge here when it comes to JS, as it is utterly enormous comapred to HTML and CSS, but a few little pointers is likely meaningful to leave here for future quick reference.

Javascript code is embedded as the content of a <script> element which is most appropriately placed at the very bottom of the <body>. Through the src attribute the actual content of the js code can be read from a separate file. For managability, this is usually the most appropriate method and allows the element content to be left empty.

<script type="text/javascript" src="file.js"> </script>
